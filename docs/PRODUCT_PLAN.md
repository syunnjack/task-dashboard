### オンライン空間にも応用できるジャンル

地図を「ページ、商品、時間」の座標に置き換えれば、同じヒートマップ部品をECやメディアにも転用できる。

| ジャンル | 可視化するもの | 収入導線 |
|---|---|---|
| EC価格・在庫 | 店舗別在庫、値下げ密度 | 商品アフィリエイト、価格通知会員 |
| 中古品・フリマ | 地域別出品と需要 | 成約、買取査定、配送 |
| ふるさと納税 | 地域・カテゴリ別人気 | 寄付成果報酬、旅行返礼品 |
| 求人 | 地域・職種・給与の需給 | 応募、スカウト課金 |
| 通信障害 | 障害報告の密度 | 回線乗換、モバイルルーター |
| SNSトレンド | 話題の地域・時間 | チケット、商品、広告分析SaaS |
| 不動産投資 | 賃料需要、空室、利回り | 面談リード、ローン、管理会社 |

最初の検証ジャンルは「都市観光×当日予約」。検索意図、地理データ、予約単価のバランスが良く、同じ基盤を飲食・ホテル・手荷物預かりへ展開できる。次点は「サウナ」「駐車場・EV充電」「コワーキング」。リアルタイム性が意思決定を変えやすく、ユーザーが混雑を避けることに金銭価値を感じやすい。

## 2. 差別化アフィリエイト設計

通常のランキング記事ではなく、`混雑を検知 → 空いている代替候補を提示 → 在庫のある予約商品へ送客` の順にする。広告枠を先に決めず、ユーザーの「待ちたくない」という意図に一致する商品だけを出す。

- コンテンツ: エリア別の混雑傾向、雨天時の代替、子連れ・一人旅などの条件別ガイド
- コンバージョン: スポット詳細の主CTA、代替候補カード、旅程保存後のまとめ予約
- 信頼: 広告表記、データ更新時刻、推定値と実測値の区別、PR順位と自然順位の分離
- 計測: `impression → map_select → alternative_view → affiliate_click → confirmed_booking` を匿名セッションで計測
- B2B: 店舗が閑散時間だけ掲載単価を上げられる「すきま枠」。成果報酬に月額の分析画面を追加

## 3. データと混雑推定

MVPは曜日・時刻・祝日・天候・イベント・ユーザーの匿名チェックインを特徴量にする。実測データが得られる施設では入退場カウンター、予約残数、Wi-Fi/Bluetooth集計などを施設同意の下で追加する。個人の位置履歴は保存せず、最低人数未満のセルは表示しない。

API例:

```json
GET /v1/crowd?bbox=139.68,35.62,139.82,35.75&at=2026-07-19T14:00:00+09:00
{
  "generatedAt": "2026-07-19T13:55:00+09:00",
  "cells": [{ "h3": "8a2f5a...", "score": 72, "confidence": 0.84 }],
  "spots": [{ "id": "spot_123", "score": 68, "waitMinutes": 20 }]
}
```

本プロトタイプの `predictedCrowd()` は、時刻とカテゴリのピークを合成するクライアント側デモ。実運用ではAPIから返る `score` と `confidence` に置き換える。

## 4. Web版

- React + Vite。地図描画は本番で MapLibre GL JS + ベクトルタイル、密度集計は H3。
- APIは TypeScript/Fastify または Cloudflare Workers、PostgreSQL/PostGIS、Redis、時系列集計基盤。
- SEO用にエリア・スポット・時間帯ガイドを静的生成。地図操作部分だけをクライアントで動かす。
- PWAでホーム画面追加、現在地周辺通知、オフライン旅程を段階導入。

## 5. アプリ版

- React Native + ExpoでiOS/Androidを共通化。Webと型定義、API SDK、混雑色、分析イベント名を共有。
- 主要タブは「マップ」「空いている」「旅程」「保存」。位置情報は利用中のみを標準にし、通知はユーザーが保存した場所に限定。
- ディープリンクで予約事業者アプリへ遷移し、戻り先を旅程画面にする。アプリストア規約に合わせ、物理サービスの外部予約として扱う。

## 6. リポジトリ案

初期はこのリポジトリでWeb MVPを検証し、実データ接続時にmonorepoへ移す。

```text
sukima-map/
  apps/web          # React / Next.js
  apps/mobile       # Expo React Native
  services/api      # crowd・spot・offer API
  packages/contracts# Zod schema / API型
  packages/analytics# 共通イベント
  packages/ui       # 色・混雑表示・カード
  infra             # DB migration / IaC
```

GitHub候補: `syunnjack/sukima-map`、`syunnjack/crowd-guide-jp`、`syunnjack/quiet-route`

## 7. ドメイン候補

- `sukima-map.jp` — 日本語で意味が伝わり、B2C/B2B両方に拡張しやすい第一候補
- `akimichi.jp` — 「空き＋道」。回遊ルート機能との相性が良い
- `komiavo.jp` — 混雑回避を短く表現したブランド名
- `quietroute.jp` — インバウンド展開向け
- `crowdless.jp` — 機能が直感的

ドメインは候補名であり、取得可能性・商標は購入直前にJPRS WHOIS、レジストラ、J-PlatPatで最終確認する。

## 8. 90日ロードマップ

1. 0–30日: 東京3エリア、公開統計＋時刻モデル、3つの予約導線でクリック率を検証
2. 31–60日: 天候・イベント・UGCチェックインを追加。代替提案のA/Bテスト
3. 61–90日: 施設10社へ「すきま枠」を提案し、予約成果と閑散時間の改善を検証

成功基準は月間利用者数より、代替候補の選択率20%、アフィリエイトCTR 8%、提携施設の閑散時間予約5件/月を先に置く。
