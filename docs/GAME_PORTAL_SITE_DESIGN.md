# GAME PORTAL JP サイト設計書

## 1. サービス概要

レトロゲームから最新ゲームまでを対象に、攻略情報と全国のゲーム大会情報を一か所で探せるポータルサイトを構築する。

サービスの二本柱は以下とする。

1. 攻略データベース
   - ゲーム別の基本情報、攻略チャート、ボス攻略、アイテム、キャラクター、対戦テクニック
   - 初心者向け・復帰勢向け・上級者向けを明確に分ける
   - レトロゲームは機種・版・地域差、最新ゲームはアップデート版を管理する
2. 大会ポータル
   - オンライン大会と実店舗・会場大会を横断検索する
   - 都道府県、開催日、ゲーム、参加条件、形式、賞品などで絞り込む
   - 主催者が大会を登録し、運営側が確認・承認できる

仮称は「GAME PORTAL JP」。正式名称とドメインはブランディング段階で決定する。

## 2. サービスコンセプト

### コアメッセージ

「遊び方が見つかる。戦う場所が見つかる。」

### 提供価値

- 昔のゲームも新作も、同じ検索体験で攻略情報へ到達できる
- 大会の開催場所・日時・参加方法を迷わず比較できる
- 店舗や小規模コミュニティも大会を告知できる
- 攻略情報から大会参加へ、大会から練習用攻略へ相互に移動できる
- 更新日・対応バージョン・情報源を表示し、古い情報を見分けられる

### 差別化方針

- 最新タイトル偏重ではなく、レトロゲームを正式な一カテゴリとして扱う
- 攻略メディアと大会カレンダーをゲームIDで統合する
- オンラインと実店舗を同じ検索画面で比較する
- 大会情報に「主催者確認済み」「公式情報確認済み」「ユーザー投稿」を表示する
- ゲームの版、機種、アップデートごとに攻略情報を管理する

## 3. 想定ユーザー

### プレイヤー

- 最新ゲームを効率よく攻略したい人

- 遠征可能な大会を地域から探したい競技プレイヤー

- eスポーツ施設、イベント会社、自治体、学校
- オンラインコミュニティ、配信者、サークル
- メーカー、パブリッシャー、公認リーグ運営者

### 攻略投稿者・編集者

- 特定タイトルに詳しいプレイヤー
- RTA・スコアアタック・対戦コミュニティ
- 攻略動画制作者、配信者、ライター
- 運営編集部と外部監修者

## 5. 主要画面設計

### 5.1 ホーム

第一画面は「ゲーム名・大会名・地域」を検索できる統合検索を中心にする。

表示順序：

1. 統合検索
2. 今週・今月の大会
3. 現在地または選択地域の大会
4. 人気ゲーム・新着ゲーム
5. レトロゲーム特集
6. 新着攻略・更新攻略
7. オンライン大会
8. 初参加向け大会ガイド
9. 主催者向け大会登録導線

### 5.2 ゲーム一覧

絞り込み：

- タイトル名
- 発売年代
- プラットフォーム
- ジャンル
- シリーズ
- メーカー
- プレイ人数
- オンライン対応
- 大会あり
- 攻略充実度

カードにはタイトル、機種、発売年、ジャンル、攻略数、今後の大会数を表示する。

### 5.3 ゲーム詳細

ゲームを攻略と大会の共通ハブにする。

- 基本情報
- 対応プラットフォーム・版
- 初心者向け攻略
- 攻略カテゴリ
- 最新更新攻略
- 開催予定大会
- 過去大会結果
- 関連ゲーム・シリーズ
- 更新履歴

レトロゲームでは移植版・リメイク版・地域版の違いを明示する。ライブサービス型ゲームでは対応パッチを表示する。

### 5.4 攻略記事

- 対象ゲーム・機種・版・パッチ
- 難易度と対象レベル
- 結論・要点
- 目次
- 手順本文
- マップ・表・チェックリスト
- 注意点、取り返しのつかない要素
- 出典・検証方法
- 著者・監修者
- 公開日・更新日・変更履歴
- 関連攻略
- 関連大会
- 誤情報の報告

ネタバレ部分は明示し、ユーザーが開くまで隠せるようにする。

### 5.5 大会検索

表示形式はリスト、カレンダー、地図の3種類とする。

絞り込み：

- ゲーム
- オンライン／実店舗／ハイブリッド
- 都道府県・市区町村
- 現在地からの距離
- 開催日・曜日・時間帯
- 初心者歓迎
- 年齢制限
- 参加費
- チーム／個人
- 使用プラットフォーム
- 公認／非公認
- 受付中／残りわずか／締切
- バリアフリー・親子参加

### 5.6 大会詳細

- 大会名・ゲーム名
- 開催日時・受付期限
- オンライン／会場情報
- 地図、最寄り駅、駐車場
- 参加費、定員、参加条件
- 大会形式、ルール、使用機種
- 必要な持ち物
- 賞品・参加特典
- 配信URL
- 主催者プロフィール
- 公式申込URL
- キャンセル・問い合わせ情報
- 情報確認ステータス
- 更新履歴
- 同ゲームの攻略リンク
- 同地域・同ゲームの関連大会

### 5.7 大会登録

入力を5段階に分ける。

1. 基本情報
2. 日時・会場・オンライン環境
3. ルール・参加条件
4. 申込・主催者情報
5. プレビュー・申請

下書き保存、複製、定期開催、開催中止、満員表示に対応する。

## 6. 検索・推薦設計

### 統合検索

同じ検索欄から以下を横断する。

- ゲーム
- 攻略
- 大会
- 主催者
- 特集記事

入力補完では、正式名称、略称、旧称、シリーズ名、機種名を扱う。

### 大会推薦

- お気に入りゲーム
- 居住地域・移動可能距離
- オンライン参加可否
- 過去に閲覧・保存した大会
- 初心者向けなどの参加属性

推薦理由を「お気に入りゲーム」「大阪府で開催」「オンライン参加可」のように表示する。

## 8. 大会情報の収集・審査

### 情報経路

1. 主催者による直接登録
2. 公式大会サイト・メーカー情報から編集部登録
3. 店舗・コミュニティの公開情報から編集部登録
4. ユーザーからの情報提供
5. 将来的な提携先フィード・API

### 信頼ステータス

- 公式主催者確認済み
- 公式URL確認済み
- 編集部確認済み
- ユーザー投稿・未確認
- 情報更新待ち
- 中止・終了

大会参加の最終判断は主催者の公式情報で行うよう、すべての詳細ページに注意書きを表示する。

### 更新運用

- 開催7日前、前日、当日に自動再確認タスクを作る
- 受付期限切れ、満員、中止を即時反映する
- 終了後は結果・配信アーカイブを追記する
- 長期間未確認の大会には警告を表示する

## 9. 攻略コンテンツの品質管理

- 対応バージョンを必須表示
- 著者と検証者を分けられる
- 重要な数値はゲーム内検証または公式情報へ紐付ける
- 更新差分を変更履歴として残す
- 誤情報報告を受け付ける
- AI生成文は編集者がゲーム内確認してから公開する
- 無断転載を禁止し、画像・動画・引用の権利を管理する
- レトロゲームは実機・移植版・エミュレーション環境の差を明記する

## 10. SEO・LLMO設計

- Game、VideoGame、Article、Event、Organization、Place、BreadcrumbList等の構造化データを表示内容と一致させる
- ゲーム、攻略、大会、地域を恒久URLで管理する
- 大会終了後も結果ページとして価値を残す
- 都道府県ページは薄い自動生成ページにせず、開催予定・会場・地域ガイドを含める
- 攻略記事は質問見出し、結論先出し、手順、FAQ、対応版、著者、更新履歴を備える
- 主催者・会場・ゲームの表記を統一する
- XMLサイトマップをゲーム、攻略、大会、記事に分ける
- 終了・中止・重複大会のcanonical、noindex、リダイレクト方針を決める

## 11. 通知・会員機能

- お気に入りゲームの新着攻略
- お気に入りゲームの大会追加
- 指定都道府県・距離内の大会
- 受付開始・締切前・開催前通知
- 大会内容変更・中止通知
- 保存した攻略の更新通知

通知手段はメールを初期実装し、Web Pushとアプリ通知は後期フェーズで追加する。

## 12. 収益化

優先順位：

1. 大会・店舗の注目掲載枠
2. ゲーム・周辺機器・店舗の広告
3. アフィリエイト
4. 主催者向け有料管理機能
5. スポンサー大会・特集
6. データ提供・API

広告と編集記事を明確に区別し、ランキングやおすすめを広告購入で不正に変更しない。

## 13. 管理画面

- 大会審査キュー
- 攻略記事の編集・レビュー
- ゲーム・版・機種マスター
- 主催者本人確認
- 会場マスターと重複統合
- 通報・権利侵害対応
- 期限切れ・未確認情報一覧
- 検索ゼロ件・需要分析
- PV、保存、申込クリック、訂正率

## 14. 非機能要件

- モバイルファースト
- Core Web Vitalsを意識した高速表示
- WCAG 2.2 AAを目標
- 地図を使えないユーザーにも住所・交通情報を提供
- 日本語を初期対象とし、多言語化可能なデータ構造にする
- 時刻はタイムゾーンを保持し、日本時間で明示する
- 個人情報を最小限にし、主催者の私的連絡先を公開しない
- 大会申込・決済は初期段階では主催者公式サイトへ遷移させる
- 投稿・編集・承認の監査ログを保持する

## 15. 推奨技術構成

### 初期版

- フロントエンド：React + Vite
- API：Node.jsまたはCloudflare Workers
- データベース：PostgreSQLまたはCloudflare D1
- 検索：PostgreSQL全文検索から開始
- 地図：Google MapsまたはMapbox
- 画像：オブジェクトストレージ + CDN
- 認証：メールリンクまたは外部認証
- 分析：GA4 + 独自イベント

### 拡張時

- 検索をAlgolia、Typesense、OpenSearch等へ移行
- 主催者フィード・API連携
- 位置情報検索
- PWA・モバイルアプリ
- 大会チェックイン・トーナメント表
- 攻略投稿者の貢献度・専門バッジ

## 16. MVP範囲

### MVPに含める

- ゲーム一覧・詳細
- 攻略一覧・詳細
- 大会一覧・詳細
- ゲーム、地域、日付、オンラインの絞り込み
- カレンダー表示
- 大会登録フォーム
- 管理者の審査・公開
- お気に入り
- メール通知

### MVPから外す

- 独自決済
- 大会トーナメント進行

## 17. 開発フェーズ

### Phase 0：検証

- 重点ゲーム20〜30タイトルを選ぶ
- 協力店舗・主催者を5〜10組確保する
- 大会情報の登録・確認フローを手作業で検証する

### Phase 1：MVP

- 攻略と大会の基本閲覧
- 検索・絞り込み
- 主催者投稿と管理審査
- お気に入りと通知

### Phase 2：成長

- 地図・現在地検索
- 投稿攻略・監修者制度
- 主催者ダッシュボード
- 大会結果・配信アーカイブ

### Phase 3：プラットフォーム化

- 外部API・データ提供
- 大会運営機能
- スポンサー・有料主催者機能
- 多言語・海外大会

## 18. 初期KPI

- 掲載ゲーム数
- 公開攻略数と更新率
- 掲載大会数
- 開催前に確認済みの大会比率
- 大会詳細から申込先へのクリック率
- 検索ゼロ件率
- お気に入り登録率
- 通知からの再訪率
- 主催者の継続登録率
- 誤情報報告率と修正時間

## 19. 次に決める項目

実装前に以下を決定する。

1. 正式なサービス名とドメイン
2. 最初に扱う重点ゲーム20〜30タイトル
3. 大会情報を集める地域・主催者パートナー
4. 攻略記事を編集部制作、投稿型、混合型のどれにするか
5. 地図サービスと検索基盤
6. 会員登録をMVPへ含めるか
7. 初期収益モデル
8. ロゴ、配色、画面密度などのビジュアル方向

## 20. 推奨する初期方針

- 攻略は編集部制作を中心にし、投稿は審査制で段階的に開放する
- 大会申込と決済は主催者公式ページへ任せ、情報ポータルに集中する
- 最新ゲームだけでなく、対戦・RTA・スコア競技が活発なレトロゲームを重点採用する
- 全国対応を掲げつつ、初期は協力主催者を確保できる地域から情報密度を高める
- 最初の成功指標はPVではなく「確認済み大会数」「申込先クリック」「情報更新率」とする
